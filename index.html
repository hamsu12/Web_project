Story

안녕하세요. 프론트엔드 개발자를 꿈꾸는 함수정입니다 !
저는 어릴 적부터 사람들과 소통하는 것을 좋아해, 반장, 학생회, 과대 등 다양한 역할을 자진해서 맡아왔습니다.
그만큼 의견을 나누고 피드백을 주고받는 과정에 익숙하며, 이를 통해 동기부여를 얻는 스타일입니다.
고등학교 1학년 정보 수업에서 처음 C언어를 접했을 때, 코드를 통해 무언가를 만들어내고,
그 결과로 다른 사람과 ‘작지만 확실한 소통’ 을 할 수 있다는 점에 큰 흥미를 느꼈습니다.

프로그래밍에 대한 흥미가 커지면서 CodeUp과 Baekjoon에서 알고리즘 문제를 풀며 기초를 다졌고,
점차 눈에 보이는 결과물을 만들고 싶어 웹 개발을 독학하게 되었습니다.
HTML과 CSS로 숫자 추측 게임을 만든 후, 친구들과 정보 선생님께 보여드렸을 때 받은 반응은 단순한 칭찬을 넘어,
“색상이 아쉬운데?”, “이 부분에서 멈칫하게 되더라”와 같은 구체적인 조언이었습니다.
정보 선생님께서는 기능적인 부분뿐 아니라 사용자의 입장에서 생각해보는 태도, 피드백을 수용하는 자세의 중요성을 알려주셨고,
그 과정에서 개발에 대한 시야와 관점을 넓힐 수 있었습니다.

이 경험은 단순히 기술적인 성장에 그치지 않았고,
‘소통하며 개선하는 과정 자체가 개발의 핵심이구나’라는 마인드적인 변화로 이어졌습니다.
더 많은 친구들과 제 결과물을 보여주고자 직접 hamsu.kr 도메인을 구매해 프로젝트를 연결하기도 했습니다.
이처럼 사용자와 소통하며 개선하는 과정이 얼마나 보람차고 재미있는지 깨닫게 된 이 순간이,
정보 선생님을 통해 성장하며, 제가 프론트엔드에 매력을 느끼게 된 결정적인 계기였습니다.

그 이후에도 고등학교 3년 내내 정보 선생님께서는 꾸준히 진로와 학습에 대해 관심을 가져주시고,
제가 어려움을 겪을 때마다 현실적인 조언과 따뜻한 격려를 아끼지 않으셨습니다.
이러한 지속적인 멘토링은 저에게 프론트엔드 개발자로서의 방향성과 자신감을 심어주는 데 큰 도움이 되었고,
더 깊이 있는 학습을 이어가는 원동력이 되었습니다.


대학생이 된 후에는 개인 프로젝트를 넘어 팀원들과 협업하는 경험을 다수 쌓았습니다.
특히 자바2 팀 프로젝트 수업에서 제작한 여행 계획 공유 플랫폼 ‘PLAN-J’는 프론트엔드와 백엔드가 함께 협업하며
GitHub를 통한 버전 관리, 피드백 기반의 기능 개선 등을 경험할 수 있었던 의미 있는 프로젝트였습니다.
이 과정에서 사용자와 직접 상호작용하는 화면을 설계하고 개선하는 일이 저에게 가장 큰 동기와 흥미를 준다는 것을 확실히 느꼈습니다.

물론 어려움도 있었습니다.
‘PLAN-J’는 처음으로 GitHub를 사용하는 팀 프로젝트 였기에,
프론트엔드와 백엔드가 협업하는 과정에서 깃 브랜치 전략이나 협업 방식에 대한 이해가 부족해 여러 번의 충돌이 발생하였습니다.
이를 해결하고자 다른 팀의 우수 사례를 분석해보며 효율적인 깃 구조와 협업 방식을 학습했습니다.
또한 여러 선배들에게 조언을 구하며 브랜치 전략, 코드 리뷰 방식, 커밋 메시지 규칙 등을 정리하고
팀원들과 공유하면서 협업 환경을 점차 개선할 수 있었습니다.
이 과정은 협업과 구조 설계의 중요성을 몸소 체감하는 계기가 되었고, 팀워크의 힘을 배울 수 있었던 소중한 경험이었습니다.



현재는 구름톤 유니브 프로젝트 ‘소담소담’을 진행 중입니다.
처음으로 리액트를 활용해 프로젝트를 진행하다 보니 컴포넌트 설계나 폴더 구조, 상태 관리 등에서 경험 부족을 절실히 느꼈고,
백엔드 팀원들과의 기술적 소통에서도 어려움을 겪었습니다.
특히 백엔드 API와 연결하는 과정에서 내가 어떤 데이터를
어떻게 활용해야 할지 헷갈리는 부분이 많았고, 그로 인해 오해가 생기기도 했습니다.
하지만 이러한 문제를 해결하고자 리액트 구조 설계에 대해 따로 공부하고,
실제로 필요한 화면이나 기능을 기준으로 컴포넌트를 리팩토링하는 연습을 반복하며 점차 나아지고 있습니다.
현재는 카카오 API를 활용한 로그인 및 위치 기반 기능을 구현 중이며, 이를 통해 기술 스택을 넓히고 있습니다.

해당 프로젝트는 기획자, 디자이너, 프론트엔드, 백엔드가 철저히 역할을 나누어 협업하는 구조로 진행되고 있기에,
단순한 기능 구현을 넘어서 API 명세서 작성, DB 설계, Git 기반 협업 등 실무 중심의 개발 프로세스를 폭넓게 경험할 수 있었습니다.
특히 각 직군과의 소통을 통해 의도 파악, 요구사항 정리, 일정 조율 등 소프트 스킬도 함께 키울 수 있었고,
개발자는 단순히 코드를 구현하는 사람이 아닌, 다양한 역할과 함께 목표를 실현해나가는 협업자임을 다시 한 번 실감했습니다.

앞으로는 단순히 보기 좋은 화면을 넘어서,
디자인과 기능성을 조화롭게 고려하여 사용자 경험(UX)을 향상시키는 인터페이스를 만드는 것이 저의 목표입니다.
사용자의 입장에서 생각하고, 더 나은 상호작용을 설계하는 프론트엔드 개발자가 되기 위해 노력하겠습니다 !

소비자를 속이는 온라인 마케팅 전략 " 다크패턴(Dark Patterns) "


다크패턴이란 ?

> 사용자의 의사결정을 의도적으로 방해하거나, 왜곡하는 UI/UX 기법


“ 탈퇴 버튼은 왜 항상 잘 안 보일까? ”

우리의 일상, 디지털 서비스 속 에는,
사용자 몰입을 유도하거나 이탈을 어렵게 만드는 교묘한 설계들이 숨어 있습니다.
그 대표적인 예가 바로 다크패턴(Dark Pattern)입니다.

다크패턴이란, 사용자의 합리적인 의사결정을 의도적으로 방해하거나 왜곡하여
기업의 이익을 극대화하려는 기만적인 UI/UX 설계 기법을 말합니다.
이러한 디자인은 사용자가 원치 않는 행동 결제를 하거나, 구독을 유지하게 만드는 일
을 하도록 유도하며, 결과적으로 소비자의 권리를 침해하는 문제가 발생합니다.

방송통신위원회와 공정거래위원회에서 분류한 다크패턴의 유형은 다음과 같습니다:
① 편취형
숨겨진 결제, 거짓 할인, 허위 추천 등
→ 사용자에게 금전적 이득이 있는 것처럼 가장해 결정을 유도하는 유형입니다.

② 오도형
잘못된 계층 구조, 특정 옵션의 사전 선택 등
→ 사용자를 헷갈리게 만들어, 실수로 특정 행동을 하게끔 설계합니다.

③ 방해형
취소·탈퇴 방해, 숨겨진 정보, 가격 비교 방해 등
→ 찾기 어렵게 만들어, 사용자가 탈퇴나 이탈을 포기하게 만듭니다.

④ 압박형
‘재고 소진 임박’, ‘지금 27명이 보고 있습니다’ 등의 감정적 문구 사용
→ 사용자에게 불안감을 조성해 구매나 가입을 서두르게 만듭니다.

실제로 일부 온라인 서비스에서는 단순한 구독 해지를 위해 평균 7번 이상의 클릭을 요구합니다.
버튼의 위치를 숨기거나, 관련 정보를 찾기 어렵게 배치하여 사용자의 의도적 행동을 방해하는 것입니다.
또한, 무료 체험 종료 후 별도 고지 없이 자동 유료 결제로 전환되거나,
“해지하시면 더 이상 혜택을 받으실 수 없습니다”와 같은 감정적 압박 문구를 통해 사용자가 해지를 망설이게 하는 경우도 있습니다.
이러한 다크패턴은 사용자의 금전적 손실, 개인정보 유출, 소비자 권리 침해로 이어질 수 있으며,
장기적으로는 디지털 환경 전체에 대한 신뢰 저하를 초래합니다.

다크패턴의 확산에 대응하여,
방송통신위원회는 『디지털 서비스 사용자 보호를 위한 다크패턴 사례집』을 발간하며 다양한 사례를 소개하고,
사용자와 기업 모두에게 경각심을 심어주고자 노력하고 있습니다.
또한 공정거래위원회는 『온라인 다크패턴 자율관리 가이드라인』 을 제정해, 기업들이 윤리적이고 책임 있는 디자인을 채택하도록 유도하고 있습니다.

기만적인 설계는 단기적으로는 사용자 행동을 유도할 수 있을지 모릅니다.
하지만 그로 인해 사용자가 피로를 느끼고, 서비스에 대한 불신이 쌓이면 결국 브랜드와 기업 모두가 외면받게 됩니다.
기업의 윤리적 책임은 선택이 아닌 필수입니다.
사용자의 시간, 주의력, 감정까지 소비하도록 설계된 인터페이스는 결국 외면받게 되어 있습니다.
진정한 경쟁력은 사용자에 대한 존중에서 시작됩니다.
그 존중은, 투명하고 정직한 디자인, 사용자 중심의 설계에서 비롯되며,
그렇기에 사람을 피로하게 만드는 디자인은 오래가지 못할거라 생각합니다.

특히 카카오 API를 활용한 지도 연동 작업을 통해 API 연동 방식에 대해 깊이 배우게 되었고, 프론트엔드와 백엔드 간의 협업 과정에서 요구사항 조율, 요청과 응답 설계, PR 메시지 작성 등 실무적인 소통 방식을 처음 경험하면서 실제 개발 현장이 어떻게 돌아가는지 직접 체감할 수 있었습니다.

또한 매일 정해진 시간에 각자 진행한 파트를 꼬박꼬박 공유하는 습관이, 팀원 간의 소통을 원활하게 하고 개발 진행 상황을 명확히 파악하는 데 큰 도움이 되었습니다.
이러한 규칙적인 커뮤니케이션은 문제 발생 시 신속한 대응을 가능하게 했고, 개발 효율과 팀워크 향상에 긍정적인 영향을 끼쳤다고 생각합니다.

이번 프로젝트를 통해 코딩 실력 향상을 비롯해 팀원과의 협업과 커뮤니케이션 능력, 그리고 전체 서비스 흐름을 이해하는 시야가 한층 넓어진 것 같습니다. 앞으로도 프로젝트 완성도를 높여가며, 다가오는 여름방학 전까지 소담소담을 성공적으로 마무리한 뒤,
구름톤 해커톤에서 좋은 성과를 얻는 것을 목표로 하고 있습니다.

1. Component 기반 구조

Component는 독립적인 단위의 소프트웨어 모듈을 말합니다.
즉, 소프트웨어를 독립적인 하나의 부품으로 만드는 방법이라고 볼 수 있습니다.

React는 UI(View)를 여러 컴포넌트(component)를 쪼개서 만듭니다.
한 페이지 내에서도 여러 각 부분을 독립된 컴포넌트로 만들고, 이 컴포넌트를 조립해 화면을 구성합니다.
컴포넌트 단위로 쪼개져 있기 때문에, 전체 코드를 파악하기가 상대적으로 쉽습니다.

기능 단위, UI 단위로 캡슐화시켜 코드를 관리하기 때문에 재사용성이 높습니다.
따라서 코드는 반복해 입력할 필요 없이, 컴포넌트만 import해 사용하면 된다는 간편함이 있으며,
애플리케이션이 복잡해지더라도 코드의 유지보수, 관리가 용이해지는 장점을 가집니다.

  
import React from 'react';
import '../App.css';


export default function Header() {
    return (
        <header className="header">
            <div className="logo">소담소담</div>
            <div className="search">
                <input type="text" placeholder="장소, 주소 등 검색" />
            </div>
        </header>
    );
}

위와 같이 Header, Footer 같은 구조 등을 컴포넌트로 만들고,
이를 조합하여 root component를 만드는 방식입니다.

2. JSX

JSX란?
JavaScript를 확장한 문법입니다.
React에서는 이벤트가 처리되는 방식, 상태(state)가 시간에 따라 변하는 방식,
데이터가 준비되어 화면에 표시되는 방식 등이 렌더링 로직과 자연스럽게 연결되곤 합니다.
그래서 React는 마크업과 로직을 인위적으로 따로 분리하지 않습니다.
대신 이 둘을 함께 담은 “컴포넌트”라는 단위를 사용하여 관심사를 분리합니다.

JSX는 꼭 사용해야 하는 것은 아니지만,
많은 사람들이 JavaScript 코드 안에서 UI 작업을 할 때, JSX가 시각적으로 더 이해하기 쉽다고 느낍니다.
또한 React가 더 유용한 에러 및 경고 메시지를 제공할 수 있도록 도와줍니다.
